# Incident: 2026-04-06 13-33-00

## Summary

Between approximately 13:33 MDT and shortly after mitigation on 2026-04-06, users attempting to place pizza orders through JWT Pizza encountered order failures. The application was healthy enough to authenticate users and create order records, but the downstream Pizza Factory fulfillment step returned HTTP 500 responses. I was alerted through an email from Grafana.

The incident was triggered by an injected chaos condition on the Pizza Factory fulfillment path. A bug in JWT Pizza was initially suspected, but logs showed successful login, successful database activity, successful factory request transmission, and a structured factory response containing `"message":"chaos monkey"` and a support `reportUrl`. The incident was detected through Grafana logs and manual traffic testing. This was a high-severity incident for order placement because successful checkout/fulfillment was unavailable during the impact window.

## Detection

This incident was detected during manual verification of production order flows and review of Grafana logs. The first strong indicator was that `POST /api/order` returned HTTP 500 while login requests continued to return 200.

Grafana logs narrowed the issue down quickly:
- `PUT /api/auth` returned 200
- database logs showed normal user lookup and auth token insertion
- `factory-request` logs were present
- `factory-response` logs were present with HTTP 500
- the response body included `"message":"chaos monkey"` and a support `reportUrl`

Time-to-detection could be improved by adding an alert on:
- `factory-response` status codes >= 500
- `POST /api/order` 5xx spikes
- order success-rate drops

## Impact

For the duration of the incident, users could log in and submit orders, but the final order fulfillment step failed. This meant customers were unable to successfully complete pizza purchases even though the service appeared partially healthy.

Observed impact included:
- authentication remained functional
- order creation path started but did not complete successfully
- downstream factory fulfillment returned 500
- end users received order failure behavior rather than successful checkout

No support ticket count was captured during this exercise, but the customer-visible symptom was failed order fulfillment.

## Response

The incident was investigated through production Grafana logs, production traffic tests, and inspection of the factory integration path.

Response actions included:
- verifying login and basic order creation still worked
- confirming the app could still reach the Pizza Factory endpoint
- isolating the failure to the downstream fulfillment step
- confirming the failure was not caused by basic authentication, menu retrieval, or user creation
- opening the Pizza Factory support `reportUrl` supplied in the failure response
- resolving the injected chaos condition through the support endpoint using the factory API key

There were no major delays once the structured factory error response was identified. The main obstacle was that the service initially appeared partially healthy, which made the issue look like an internal JWT Pizza bug before logs clearly showed downstream factory failure.

## Root cause

An injected chaos condition in the downstream Pizza Factory fulfillment path caused the factory to return HTTP 500 responses for order fulfillment requests. JWT Pizza itself remained healthy enough to authenticate users, create order records, and send factory requests, but fulfillment failed because the downstream dependency was intentionally disrupted.

## Resolution

The service was restored by following the `reportUrl` returned in the Pizza Factory failure response and resolving the injected chaos condition using the factory API key. The support endpoint returned:

`Problem resolved. Pizza is back on the menu!`

This confirmed that the downstream disruption had been removed. The correct recovery action was not a JWT Pizza code change, but resolving the factory-side chaos condition.

## Prevention

This class of incident can happen again whenever a downstream dependency appears reachable but is failing at business-logic execution time. The main prevention opportunities are:

- add alerts specifically for `factory-response` 5xx events
- add alerts on `/api/order` failure-rate increases
- document the meaning of the Pizza Factory `"chaos monkey"` response
- document the support `reportUrl` recovery workflow
- add a dashboard panel focused on factory request/response outcomes

## Action items

1. Add a Grafana alert for `factory-response` HTTP 5xx events
2. Add a Grafana alert for spikes in `POST /api/order` failures
3. Add a dashboard panel showing factory request/response success vs failure
4. Document the Pizza Factory support `reportUrl` recovery procedure
5. Keep factory request/response logs in place for future incident triage
6. Validate recovery by running a successful production order flow after mitigation
